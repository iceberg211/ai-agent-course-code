# 简易版 Manus 学习知识框架

## 1. 目标

这份笔记不是为了直接开做项目，而是为了先建立一个稳定的知识框架。

目标有三个：

1. 明确简易版 Manus 到底由哪些技术模块组成。
2. 判断当前课程源码已经覆盖了什么，还缺什么。
3. 给出一个从基础到项目的学习顺序，避免只是在堆功能。

当前选定的参考项目：

1. `vercel-labs/coding-agent-template`
2. `vercel/chatbot`

这两个项目的作用不同：

1. `vercel-labs/coding-agent-template` 重点看任务执行链路、Agent 运行环境、步骤化执行、产物管理。
2. `vercel/chatbot` 重点看前端聊天交互、流式输出、消息结构、用户体验组织方式。

## 2. 简易版 Manus 的本质

简易版 Manus 不是一个普通聊天机器人，它至少要同时具备下面几种能力：

1. 能理解复杂任务，而不是只回答一句话。
2. 能先规划，再执行，而不是只做单轮工具调用。
3. 能调用外部工具完成任务。
4. 能把中间步骤、工具结果、最终产物展示给用户。
5. 任务失败后可以继续，而不是整轮报废。

从产品形态上看，它更接近：

1. `任务型 Agent`
2. `可观察的执行系统`
3. `带状态的前后端应用`

而不是：

1. 纯 Prompt Demo
2. 单轮问答助手
3. 只有工具调用的脚本集合

## 3. 知识框架总览

可以把简易版 Manus 拆成 6 层。

### 3.1 模型与推理层

这一层解决的是：模型怎么理解任务，怎么输出稳定结果。

需要掌握：

1. 提示词设计
2. 结构化输出
3. Tool Calling
4. 流式输出
5. 上下文窗口管理
6. 重试与兜底

需要达到的结果：

1. 模型能够稳定输出计划、步骤、工具参数、最终结果。
2. 模型出错时，系统可以识别错误并重试，而不是直接崩。

当前仓库已有基础：

1. `prompt-template-test`
2. `output-parser-test`
3. `tool-test`
4. `runnable-test`

### 3.2 Agent 编排层

这一层解决的是：一个复杂任务怎么拆开并持续执行。

需要掌握：

1. ReAct 模式
2. Planner / Executor 拆分
3. 状态机与节点编排
4. 人工介入
5. 检查点与恢复
6. 多轮任务状态更新

这是你当前最需要补的一层，重点是：

1. `LangGraph.js`
2. 状态图思维
3. 长任务执行设计

需要达到的结果：

1. 一个任务可以从 `接收 -> 规划 -> 执行 -> 审核 -> 结束` 完整流转。
2. 中间某一步失败时，可以从失败点继续。

### 3.3 工具系统层

这一层解决的是：Agent 可以做什么，以及怎么安全地做。

需要掌握：

1. 工具定义方式
2. 工具参数校验
3. 工具权限边界
4. 工具失败处理
5. 工具结果标准化
6. 浏览器工具、文件工具、命令工具、搜索工具

简易版 Manus 的首批工具建议只保留：

1. 网页搜索
2. 浏览器访问与提取
3. 文件读取
4. 文件写入
5. 命令执行

先不要做：

1. 邮件发送
2. 外部平台写操作
3. 高风险系统指令
4. 太多第三方集成

当前仓库已有基础：

1. `tool-test`
2. `cron-job-tool`
3. `agui-backend`

### 3.4 记忆与数据层

这一层解决的是：任务历史、会话上下文、产物内容存到哪里。

需要掌握：

1. 短期记忆
2. 长期记忆
3. 向量检索
4. 任务持久化
5. 执行日志持久化
6. 产物存储

这里要区分三种东西：

1. `聊天记忆`：用户说过什么。
2. `任务状态`：现在执行到哪一步。
3. `任务产物`：最后写出来的文件、总结、报告。

当前仓库已有基础：

1. `memory-test`
2. `rag-test`
3. `milvus-test`

当前缺口：

1. 数据模型没有真正统一起来。
2. 还没有任务表、步骤表、产物表这样的项目化设计。

### 3.5 前后端交互层

这一层解决的是：用户如何发任务，如何看到执行过程。

需要掌握：

1. 流式响应
2. 前端消息渲染
3. 中间步骤展示
4. 工具调用结果展示
5. 任务详情页
6. 执行日志回放

简易版 Manus 前端至少要有：

1. 任务输入区
2. 当前计划展示区
3. 当前步骤展示区
4. 工具调用日志区
5. 最终产物展示区

当前仓库已有基础：

1. `hello-nest-langchain` 的 SSE
2. `agui-backend`
3. `agui-frontend`

当前缺口：

1. 现在更像聊天界面，不像任务执行界面。
2. 还没有任务视角的状态展示。

### 3.6 工程化与上线层

这一层解决的是：项目怎么稳定运行，怎么上线。

需要掌握：

1. 环境变量管理
2. 日志与监控
3. 错误处理
4. 鉴权
5. 部署
6. 成本控制
7. 限流与安全

这是当前仓库最薄的一层。

目前已经有一些基础：

1. Nest 服务拆分
2. MySQL 与 TypeORM
3. 定时任务

但明显还缺：

1. Docker
2. CI
3. 监控
4. 鉴权
5. 生产环境部署策略

## 4. 做简易版 Manus 必须补的知识盲点

下面这些是当前最关键的盲点，优先级从高到低排序。

### 4.1 LangGraph.js

优先级：`最高`

为什么必须补：

1. Manus 类项目不是单条链，而是状态流转系统。
2. 你后面还要讲 LangGraph，这个项目天然适合用它来做主线。

必须掌握的点：

1. 节点
2. 边
3. 状态定义
4. 条件分支
5. 子图
6. 检查点
7. 人工介入

学习结果：

1. 能把任务执行过程画成状态图。
2. 能把状态图变成代码。

### 4.2 任务系统设计

优先级：`最高`

为什么必须补：

1. 聊天消息不是任务系统。
2. Manus 的核心是任务对象，而不是单轮对话对象。

必须掌握的点：

1. 任务实体设计
2. 步骤实体设计
3. 产物实体设计
4. 运行状态枚举
5. 重试机制
6. 手动继续执行

学习结果：

1. 能设计最小任务表结构。
2. 能把前端界面改成“任务视角”。

### 4.3 浏览器工具链

优先级：`高`

为什么必须补：

1. 没有浏览器工具，Manus 的执行力会很弱。
2. 只靠 web search 不足以体现项目价值。

必须掌握的点：

1. Playwright 基础
2. 页面导航
3. 元素定位
4. 内容提取
5. 下载与截图
6. 失败恢复

学习结果：

1. 能把浏览器能力封装成 Agent 工具。
2. 能处理基本页面自动化场景。

### 4.4 流式任务可视化

优先级：`高`

为什么必须补：

1. Manus 不是只把最终答案吐出来。
2. 用户需要看到 Agent 现在在做什么。

必须掌握的点：

1. 服务端事件流
2. UI Message 结构
3. 步骤状态更新
4. 工具调用回显
5. 前端分区展示

学习结果：

1. 用户可以实时看到任务进度。
2. 出错时能定位在哪一步失败。

### 4.5 持久化与恢复

优先级：`高`

为什么必须补：

1. 任务一旦变长，就不能只放在内存里。
2. 长任务没有恢复能力，产品就不稳定。

必须掌握的点：

1. 任务状态落库
2. 步骤日志落库
3. 失败重试
4. 重启恢复
5. 产物目录管理

学习结果：

1. 服务重启后任务记录还在。
2. 用户可以回看历史任务。

### 4.6 工具安全边界

优先级：`中高`

为什么必须补：

1. Agent 一旦有文件和命令执行能力，就必须控制边界。
2. 课程如果不讲这块，后面项目会很危险。

必须掌握的点：

1. 允许执行的命令范围
2. 允许读写的目录范围
3. 高风险操作确认
4. 超时控制
5. 工具输出截断

学习结果：

1. 项目不会因为工具太强而失控。

## 5. 当前课程源码可以复用的部分

这些内容可以直接成为简易版 Manus 的前置章节材料。

### 5.1 可以直接复用的能力

1. Prompt 与模板：`prompt-template-test`
2. 结构化输出：`output-parser-test`
3. Tool Calling：`tool-test`
4. Runnable 组合：`runnable-test`
5. Memory：`memory-test`
6. RAG：`rag-test`、`milvus-test`
7. Nest 流式接口：`hello-nest-langchain`
8. 前后端流式聊天：`agui-backend`、`agui-frontend`
9. 任务执行循环示例：`cron-job-tool`

### 5.2 不能直接拿来当 Manus 项目的地方

1. 多数 `.mjs` 示例偏实验性质，缺少统一工程结构。
2. 现在的聊天界面不等于任务执行界面。
3. 现在的工具系统还没有做权限边界和产物系统。
4. 现在还没有 LangGraph.js 的主线实现。

## 6. 简易版 Manus 的最小实现边界

为了避免项目过大，第一版建议只做下面这些。

### 6.1 必做能力

1. 用户输入一个复杂任务。
2. 系统先输出任务计划。
3. 系统逐步执行计划。
4. 每一步可以调用工具。
5. 前端实时显示步骤和日志。
6. 最后生成一个结果页或 Markdown 文件。
7. 所有任务与步骤可以查询历史记录。

### 6.2 第一版只做这些工具

1. `web_search`
2. `browser_navigate`
3. `browser_extract`
4. `file_read`
5. `file_write`

### 6.3 第一版先不做

1. 多智能体
2. 代码仓库自动修改
3. 数字人
4. 语音输入输出
5. 云端沙箱
6. 超复杂权限系统
7. 第三方办公系统集成

## 7. 学习顺序

建议按下面顺序补知识。

### 第一阶段：把 Agent 基础打牢

目标：能做单任务、多步工具调用的最小循环。

学习内容：

1. Prompt
2. Structured Output
3. Tool Calling
4. Runnable
5. 错误处理

对应仓库基础：

1. `prompt-template-test`
2. `output-parser-test`
3. `tool-test`
4. `runnable-test`

### 第二阶段：补 LangGraph.js

目标：把“链”升级成“状态图”。

学习内容：

1. 状态设计
2. 节点与边
3. Planner / Executor
4. 分支流转
5. Checkpoint
6. Human in the loop

这一阶段结束后，应该能回答两个问题：

1. 为什么聊天链做不好 Manus。
2. 为什么任务系统要有状态图。

### 第三阶段：补任务系统与持久化

目标：让 Agent 能作为产品运行，而不是只在控制台里跑。

学习内容：

1. 任务表设计
2. 步骤表设计
3. 产物表设计
4. 任务状态机
5. 执行日志
6. MySQL / TypeORM 落地

### 第四阶段：补浏览器工具链

目标：让项目真正具备执行力。

学习内容：

1. Playwright
2. 浏览器动作封装
3. 网页信息抽取
4. 工具失败处理
5. 截图与调试

### 第五阶段：补前端任务执行界面

目标：让用户真正感知到 Agent 在工作。

学习内容：

1. 任务列表
2. 任务详情
3. 步骤流展示
4. 工具日志展示
5. 最终产物展示
6. 流式 UI 更新

### 第六阶段：补工程化

目标：让项目能上线演示。

学习内容：

1. 环境变量管理
2. Docker
3. 部署
4. 日志
5. 限流
6. 安全边界

## 8. 看参考项目时要重点看什么

### 8.1 看 `vercel-labs/coding-agent-template`

重点不是看 UI 漂不漂亮，而是看下面这些问题：

1. 任务对象怎么定义。
2. Agent 执行过程怎么组织。
3. 沙箱或执行环境怎么接入。
4. 中间步骤如何回传给前端。
5. 最终产物如何保存。

### 8.2 看 `vercel/chatbot`

重点不是看聊天功能本身，而是看：

1. 流式消息结构怎么设计。
2. 前端如何消费流式响应。
3. 聊天界面怎么做基础体验。
4. 如何把通用聊天 UI 改造成任务执行 UI。

## 9. 最终要形成的能力地图

学完之后，你应该具备下面这张能力地图。

### 9.1 认知层

1. 知道 Agent 不等于聊天机器人。
2. 知道 Manus 的核心是任务执行，不是对话生成。
3. 知道 LangGraph 适合解决状态编排问题。

### 9.2 开发层

1. 能定义 Agent 状态。
2. 能实现工具调用循环。
3. 能实现任务持久化。
4. 能做流式前后端交互。
5. 能接入浏览器工具。

### 9.3 项目层

1. 能做一个最小可演示的简易版 Manus。
2. 能继续演化成更完整的 Agent 产品。
3. 能把这个项目拆成课程章节，而不是一次写完。

## 10. 当前结论

一句话总结：

你现在不是不会做 Agent，而是 `Agent 单点能力有了，但缺少“任务系统 + 状态编排 + 工程化”这三块中间层`。

所以接下来的主线应该是：

1. 先补 `LangGraph.js`
2. 再补 `任务系统设计`
3. 再补 `浏览器工具链`
4. 然后正式进入 `简易版 Manus`

这条路线最稳，也最适合做成课程主线。
