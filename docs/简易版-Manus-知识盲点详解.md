# 简易版 Manus 知识盲点详解

## 1. 先说结论：你现在缺的不是“知道名词”，而是缺中间层

从当前仓库来看，你已经有这些能力：

1. 会写 Prompt。
2. 会做结构化输出。
3. 会做 Tool Calling。
4. 会做基本 RAG。
5. 会做 Memory。
6. 会做流式输出。
7. 会用 Nest 写后端。
8. 会写一个基础聊天前端。

这些能力说明一件事：

你已经具备了 `Agent 零件级知识`。

但简易版 Manus 不是把这些零件摆在一起就行，它真正依赖的是下面这层能力：

1. 怎么把零件组织成一个持续运行的任务系统。
2. 怎么让任务有状态，而不是只是一轮对话。
3. 怎么让 Agent 的执行过程能被观察、能暂停、能恢复。
4. 怎么让工具调用变成产品能力，而不是演示脚本。

这层能力，我把它叫做 `Agent 中间层`。

你当前最大的知识盲点，几乎都集中在这里。

## 2. 盲点一：把聊天式 Agent 当成任务型 Agent

这是最常见、也是最核心的误区。

### 2.1 聊天式 Agent 是什么

聊天式 Agent 的运行方式通常是这样的：

1. 用户发一句话。
2. 模型理解问题。
3. 模型可能调用几个工具。
4. 返回一段答案。

这类系统的中心对象是：

1. `消息`
2. `上下文`
3. `回答`

你当前仓库里很多示例都属于这个层面，比如：

1. [ai.service.ts](/Users/hewei/Documents/GitHub/ai-agent-course-code/hello-nest-langchain/src/ai/ai.service.ts)
2. [ai.service.ts](/Users/hewei/Documents/GitHub/ai-agent-course-code/agui-backend/src/ai/ai.service.ts)

这些代码没有问题，但它们解决的是“问答系统”问题。

### 2.2 任务型 Agent 是什么

任务型 Agent 的运行方式不一样：

1. 用户给的是一个任务，不是一个问题。
2. 系统要先判断任务目标。
3. 系统要拆步骤。
4. 系统要逐步执行。
5. 中间可能失败。
6. 失败后要继续。
7. 最后要交付结果产物。

这类系统的中心对象变成了：

1. `任务`
2. `计划`
3. `步骤`
4. `执行状态`
5. `产物`

这就是为什么聊天界面不能直接等于 Manus。

### 2.3 你现在真正该建立的认知

你要把脑子里的核心对象，从：

1. `message`

切换到：

1. `task`
2. `run`
3. `step`
4. `artifact`

如果这一点不切换，后面无论你接 LangGraph、MCP、Playwright，最后都会回到“一个会调用工具的聊天机器人”。

### 2.4 课程里应该怎么讲这块

这一部分不能只讲概念，必须让学员真正看出区别。

建议你把这一节讲清楚三个对比：

1. 聊天问题：`帮我解释一下 React Compiler 是什么`
2. 任务问题：`帮我查 5 篇 React Compiler 资料，提炼重点，生成一份笔记`
3. 系统差异：前者是回答，后者是执行

这一节如果讲透，后面的 Manus 主线就不会跑偏。

## 3. 盲点二：为什么 LangChain 不够，必须上 LangGraph

这不是说 LangChain 不好，而是它解决的问题边界不同。

### 3.1 LangChain 更适合什么

LangChain 更适合这些事情：

1. 串联 Prompt、Model、Parser。
2. 封装工具。
3. 组合 Runnable。
4. 快速搭一个 Agent Demo。

你当前仓库的这部分已经不少了：

1. `prompt-template-test`
2. `output-parser-test`
3. `tool-test`
4. `runnable-test`

这些都属于 Agent 的组件层。

### 3.2 为什么做 Manus 时会卡住

因为 Manus 不是“一条链”，而是“一个状态流”。

你迟早会遇到这些问题：

1. 当前是在规划阶段，还是执行阶段？
2. 这个步骤失败了，是重试，还是回到重新规划？
3. 用户中途修改目标后，原计划还算不算有效？
4. 当前任务执行到第几步？
5. 服务重启后，任务怎么接着跑？
6. 某个步骤需要人工确认，系统怎么暂停？

这些问题如果只靠 Runnable 组合，代码会越来越乱。

### 3.3 LangGraph 补的到底是什么

LangGraph 补的不是“更高级的 Prompt”，而是 `可控的执行结构`。

你可以把它理解成：

1. LangChain 更像函数和组件。
2. LangGraph 更像工作流引擎和状态机。

LangGraph 真正重要的价值在这几个点：

1. 有明确状态。
2. 有明确节点。
3. 有明确流转边。
4. 可以做条件分支。
5. 可以做中断与恢复。
6. 可以做人工介入。
7. 可以做持久化检查点。

### 3.4 这一块你要补到什么程度才算够

不是会跑官方 Demo 就够了，而是你要能自己回答下面这些问题：

1. 我的状态对象长什么样？
2. 哪些字段由 Planner 改，哪些字段由 Executor 改？
3. 哪个节点负责决定“下一步做什么”？
4. 哪个节点负责执行工具？
5. 失败时回到哪个节点？
6. 哪些步骤可以跳过？
7. 哪些步骤需要人工确认？

如果这些问题说不清，LangGraph 就还只是“看过”。

### 3.5 你应该建立的最小 LangGraph 心智模型

最小模型就是下面这 6 个节点：

1. `receive_task`
2. `plan_task`
3. `execute_step`
4. `review_step_result`
5. `replan_if_needed`
6. `finish_task`

注意重点：

1. `计划` 不是一次性写死的。
2. `执行` 不是一直往前跑的。
3. `复盘与重规划` 是必须存在的。

真正的 Agent 系统，核心不是“能不能调用工具”，而是“调用完工具之后，系统能不能决定下一步”。

## 4. 盲点三：不会设计任务系统

这是做 Agent 产品时最容易忽视的一块。

### 4.1 为什么任务系统比 Prompt 更重要

因为 Prompt 决定“模型怎么说”，任务系统决定“产品怎么活”。

如果没有任务系统，你会遇到这些问题：

1. 前端不知道该展示什么。
2. 后端不知道任务现在跑到哪。
3. 用户不知道失败在哪一步。
4. 历史记录无法复盘。
5. 结果产物无处可放。

### 4.2 你至少要有哪几个实体

第一版不要复杂化，最少设计 4 个核心实体。

#### Task

代表一个用户任务。

建议字段：

1. `id`
2. `title`
3. `userInput`
4. `status`
5. `currentStepIndex`
6. `createdAt`
7. `updatedAt`
8. `errorMessage`

#### TaskPlan

代表任务规划结果。

建议字段：

1. `id`
2. `taskId`
3. `version`
4. `goal`
5. `stepsJson`
6. `createdAt`

这里要注意：

1. 计划是允许重生成的，所以要有 `version`。

#### TaskStepRun

代表某一个步骤的一次执行记录。

建议字段：

1. `id`
2. `taskId`
3. `planVersion`
4. `stepIndex`
5. `stepTitle`
6. `status`
7. `toolCallsJson`
8. `resultSummary`
9. `startedAt`
10. `finishedAt`

这里要注意：

1. 步骤不只是定义，还要有运行记录。
2. 同一个步骤可能重试多次，所以最好是 `step run`，不是单纯 `step`。

#### Artifact

代表任务最终产物或中间产物。

建议字段：

1. `id`
2. `taskId`
3. `type`
4. `title`
5. `content`
6. `filePath`
7. `createdAt`

### 4.3 状态设计是任务系统的核心

任务状态最少建议这样分：

1. `pending`
2. `planning`
3. `running`
4. `waiting_human`
5. `succeeded`
6. `failed`
7. `cancelled`

步骤状态最少建议这样分：

1. `pending`
2. `running`
3. `succeeded`
4. `failed`
5. `skipped`

### 4.4 很多人为什么会把这块做烂

常见错误有 4 个：

1. 用消息表代替任务表。
2. 只存最终答案，不存过程。
3. 没有计划版本。
4. 步骤失败后没有重试语义。

### 4.5 你现在就该建立的产品意识

Manus 项目不应该以“对话记录”为核心，而应该以“任务详情页”为核心。

聊天只是入口。
任务系统才是骨架。

## 5. 盲点四：浏览器工具链理解不够

如果你要做一个像 Manus 的系统，没有浏览器工具，执行力就会差很多。

### 5.1 为什么浏览器工具很关键

因为很多真实任务不是“回答问题”，而是：

1. 打开网页。
2. 查找内容。
3. 点开链接。
4. 提取信息。
5. 整理结果。

这类任务用 `web_search` 只能做到一半。

### 5.2 浏览器工具不要一开始就做成“大而全”

第一版只做下面几类动作就够了：

1. `navigate(url)`
2. `extract_text(selector?)`
3. `click(selector)`
4. `type(selector, value)`
5. `screenshot()`

如果第一版上来就搞：

1. 复杂 DOM 语义理解
2. 自主网页探索
3. 多标签页复杂同步
4. 文件上传下载闭环

很容易做偏。

### 5.3 这块真正难的不是“会用 Playwright”

而是下面这几个问题：

1. 工具返回什么结果，模型才能继续推理？
2. 页面失败时怎么告诉 Agent 现在失败了什么？
3. 哪些动作应该由模型决定，哪些动作应该由程序兜底？
4. 如何避免 Agent 一直重复点击和重复导航？

### 5.4 工具返回值应该长什么样

不要只返回一句“操作成功”。

更好的返回值应该包括：

1. 当前页面标题
2. 当前 URL
3. 抽取到的核心文本
4. 可继续操作的候选元素
5. 错误信息

本质上，浏览器工具不是“机械执行器”，它还是下一步推理的输入源。

### 5.5 课程里怎么讲这块才不会飘

不要从“AI 自动点网页很酷”开始讲。

要从下面这个问题开始讲：

`为什么搜索结果页不是最终答案，而只是任务执行中的一个中间环境。`

这样学员就不会把浏览器工具理解成一个花哨插件，而会把它理解成任务执行环境。

## 6. 盲点五：流式输出会了，但不会做“流式任务界面”

你现在已经有流式输出基础，这点很好。

### 6.1 你现在会的流式输出是什么

更偏向：

1. 模型 token 一边生成，前端一边显示。

这属于“流式回答”。

### 6.2 Manus 需要的是什么

Manus 需要的是“流式执行可视化”。

它不只是流式文本，而是流式事件。

事件至少包括：

1. 任务已创建
2. 计划已生成
3. 当前执行第几步
4. 调用了什么工具
5. 工具返回了什么
6. 当前步骤成功或失败
7. 任务结束

### 6.3 你应该怎么理解前端

聊天前端的核心是：

1. 展示消息

任务前端的核心是：

1. 展示状态变化

所以简易版 Manus 的前端不是只需要一个聊天框，而是至少要有这几块：

1. 任务输入区
2. 当前计划区
3. 步骤执行区
4. 工具日志区
5. 结果产物区

### 6.4 为什么很多 Agent 产品看起来“不像在工作”

因为它们只把最终文本流出来，没有把过程结构化。

结果就是：

1. 用户不知道系统在干嘛。
2. 一旦卡住，用户不知道卡在哪里。
3. 产品价值看起来和普通聊天机器人差不多。

### 6.5 你真正该补的，是事件设计

你至少要设计一套事件类型。

例如：

1. `task.created`
2. `task.planned`
3. `step.started`
4. `tool.called`
5. `tool.completed`
6. `step.completed`
7. `task.completed`
8. `task.failed`

学会这件事之后，你才能把流式能力从“文字效果”升级成“产品能力”。

## 7. 盲点六：会做 Memory，不等于会做任务持久化

这两个事情经常被混在一起，但它们其实完全不是一个层级。

### 7.1 Memory 是什么

Memory 更关心：

1. 对话历史
2. 用户偏好
3. 历史上下文
4. 可检索记忆

它解决的是：

`模型需要补充哪些上下文`

### 7.2 任务持久化是什么

任务持久化更关心：

1. 当前任务状态
2. 当前跑到哪一步
3. 哪一步失败了
4. 哪些产物已经生成
5. 能不能恢复运行

它解决的是：

`系统现在运行到哪里`

### 7.3 为什么这两个不能混

如果你把任务状态直接塞进聊天记忆里，会出现这些问题：

1. 状态不可控
2. 很难查询
3. 很难回放
4. 很难做恢复
5. 很难做统计

所以你必须建立一个非常明确的边界：

1. `Memory 给模型看`
2. `Task State 给系统用`

### 7.4 你应该怎么补这一块

你现在最该补的不是“再学一种记忆算法”，而是：

1. 学会把任务运行状态落库。
2. 学会从数据库恢复任务。
3. 学会让前端根据数据库状态渲染任务详情。

## 8. 盲点七：不会处理失败、重试和人工介入

这块其实是任务型 Agent 和 Demo Agent 的分水岭。

### 8.1 为什么 Demo 都很顺

因为 Demo 默认：

1. 网络正常
2. 页面正常
3. 工具正常
4. 模型理解正确
5. 不需要人工确认

现实情况不是这样。

### 8.2 简易版 Manus 至少要考虑的失败类型

1. 模型输出不合法
2. 工具参数不合法
3. 网页打不开
4. 目标页面元素不存在
5. 工具调用超时
6. 步骤执行结果不符合预期

### 8.3 重试不能是“统一来一遍”

你要区分：

1. `模型重试`
2. `工具重试`
3. `步骤重试`
4. `重新规划`

这是四种不同层级的重试。

比如：

1. JSON 解析失败，适合模型重试。
2. 网页短暂超时，适合工具重试。
3. 当前步骤结果不好，适合步骤重试。
4. 当前计划本身错了，适合重新规划。

### 8.4 人工介入为什么必须讲

因为很多真实任务不是“系统永远自己跑完”。

常见需要人工确认的点：

1. 是否继续执行高风险操作
2. 当前结果是否满意
3. 是否要重规划
4. 是否要提交最终产物

一旦你讲 LangGraph，就必须顺手把这一块带进去。

不然学员很容易误以为 Agent 就是“自动一直跑”。

## 9. 盲点八：工具安全边界意识不足

这是做 Agent 项目时最容易出事故的地方。

### 9.1 为什么这块不是可选项

因为一旦 Agent 有这些工具：

1. 文件读写
2. 命令执行
3. 浏览器操作
4. 外部接口调用

它就已经不是普通聊天机器人了。

这时候最少要建立 4 个安全边界。

### 9.2 目录边界

例如：

1. 只允许读写工作目录。
2. 不允许访问系统敏感目录。
3. 不允许覆盖关键配置文件。

### 9.3 命令边界

例如：

1. 命令白名单
2. 禁止高危命令
3. 超时自动终止
4. 输出长度限制

### 9.4 操作边界

例如：

1. 写操作需要确认
2. 外部请求有预算限制
3. 浏览器动作有步数限制

### 9.5 成本边界

例如：

1. 最大循环次数
2. 最大工具调用次数
3. 最大上下文长度
4. 最大任务运行时长

这部分课程里如果不讲，学员做出来的 Agent 基本都只能算“演示”，不能算“可交付系统”。

## 10. 盲点九：工程化意识还没有真正进入 Agent 项目

这一块不是最酷，但一定是最后最卡人的地方。

### 10.1 你以后一定会遇到的问题

1. 环境变量怎么组织
2. 不同模型怎么切换
3. 工具日志怎么查
4. 用户怎么登录
5. 项目怎么部署
6. 怎么避免接口被刷
7. 怎么知道哪一步最耗时

### 10.2 简易版 Manus 第一版最少要做什么

你不用一上来就企业级，但最少要有：

1. `.env` 管理
2. 请求日志
3. 错误日志
4. 基础鉴权
5. Docker 化
6. 基础部署说明

### 10.3 为什么这块也算知识盲点

因为很多 Agent 教程都在讲模型、工具、Prompt，但真实项目最后花时间最多的，往往是这些配套部分。

你如果后面课程要讲“部署上线”，那这块不能后知后觉地补。

## 11. 把这些盲点映射成学习任务

如果你要真正补知识，不要再写“学习 LangGraph”“学习 Playwright”这种泛目标。

要改成可验收的学习任务。

### 11.1 LangGraph 学习任务

你学完之后，必须能做到：

1. 画出简易版 Manus 的状态图。
2. 用代码实现 `规划 -> 执行 -> 重规划 -> 结束`。
3. 给任务加一个人工确认节点。
4. 给任务加一个失败重试分支。

### 11.2 任务系统学习任务

你学完之后，必须能做到：

1. 设计任务表、步骤表、产物表。
2. 定义任务状态与步骤状态。
3. 能从数据库查询一个任务的完整执行轨迹。
4. 前端能展示任务详情页。

### 11.3 浏览器工具学习任务

你学完之后，必须能做到：

1. 封装最小浏览器工具。
2. 支持页面导航、信息抽取、截图。
3. 工具失败时返回结构化错误。
4. Agent 能根据工具返回值决定下一步。

### 11.4 流式界面学习任务

你学完之后，必须能做到：

1. 设计事件类型。
2. 服务端流式推送任务事件。
3. 前端分区展示计划、步骤、日志、产物。
4. 任务失败时能明确标出失败点。

### 11.5 工程化学习任务

你学完之后，必须能做到：

1. 项目能本地一键启动。
2. 项目能 Docker 运行。
3. 项目关键日志可以追踪。
4. 高风险工具操作有边界控制。

## 12. 你现在最该补的顺序

如果按照“对简易版 Manus 的贡献度”排序，我建议是：

1. `LangGraph.js`
2. `任务系统设计`
3. `流式任务事件与前端任务界面`
4. `浏览器工具链`
5. `任务持久化与恢复`
6. `工具安全边界`
7. `工程化与部署`

这个顺序不是按“哪个技术更火”，而是按“哪个技术最决定项目能不能成立”。

## 13. 最后再压缩成一句话

你现在的短板不是模型能力，不是 Prompt 能力，也不是 LangChain 基础。

你真正缺的是：

`怎么把 Agent 从“会说、会调工具”提升成“会执行任务、可观察、可恢复、可交付”的系统能力。`

这就是简易版 Manus 里最该先补的知识盲点。
