# 炒股 Agent 系统 — 技术方案

## 1. 系统目标

基于 Deep Agents 框架，构建一个股票交易辅助系统。系统能够自动收集市场信息、综合分析、生成交易决策，并将结果推送给用户。

核心设计原则：
- **信息收集分散，决策集中** — 多个 Subagent 并行收集数据，但只有一个 Decision Agent 做判断
- **硬的用 Tool 约束，软的用 Prompt 引导** — 风控规则不可绕过，交易判断靠推理
- **策略可插拔** — 新增策略不改代码，只加 Skill 文件

## 2. 整体架构

### 2.1 四类角色

| 角色 | 职责 | 不做什么 |
|------|------|----------|
| Orchestrator | 接收用户请求，拆任务，派活，调度流程 | 不分析、不决策 |
| 信息收集 Subagent (x3) | 技术面/基本面/舆情，返回结构化事实数据 | 不给结论、不给建议 |
| Decision Agent | 综合所有信息，在策略框架下做出交易决策 | 不自己收集数据 |
| 策略库 (Skills) | 定义不同行情下的交易规则和参数 | 不包含代码逻辑 |

### 2.2 信息流

```
用户请求
  |
  v
Orchestrator（判断需要收集什么信息）
  |
  +---> 技术面 Subagent（并行）---> 返回事实数据
  +---> 基本面 Subagent（并行）---> 返回事实数据
  +---> 舆情 Subagent（并行）-----> 返回事实数据
  |
  v
Orchestrator（汇总信息，交给 Decision Agent）
  |
  v
Decision Agent（四步管线决策）
  |
  v
推送交易信号 / 自动下单（HITL）
```

### 2.3 Deep Agents 框架映射

| 框架能力 | 用在哪里 |
|----------|----------|
| `create_deep_agent()` | 创建 Orchestrator 主 Agent |
| `SubAgentMiddleware` | 技术面/基本面/舆情 Subagent + Decision Agent |
| `TodoListMiddleware` | Orchestrator 拆解分析步骤 |
| `SkillsMiddleware` | 策略库按需加载 |
| `HumanInTheLoopMiddleware` | 自动下单模式的 `place_order` 审批 |
| `Store` | 跨会话持久化：用户风险偏好、持仓状态、交易记录 |
| `FilesystemBackend` | 策略 Skill 文件的读取 |

## 3. 信息收集 Subagent

三个 Subagent 职责相同：调用 Tool 获取数据，按结构化格式返回，不做判断。

### 3.1 技术面 Subagent

**输入**：股票代码、时间周期

**可用 Tools**：
- `get_kline` — 获取 K 线数据
- `calc_indicators` — 计算技术指标（MACD、RSI、KDJ、布林带等）
- `detect_patterns` — 识别 K 线形态

**输出格式**：
- 当前价格、均线位置（5/10/20/60日）
- 各技术指标数值
- 成交量与均量对比
- K 线形态识别结果
- 关键支撑位和阻力位

### 3.2 基本面 Subagent

**输入**：股票代码

**可用 Tools**：
- `get_financials` — 获取财报数据
- `get_valuation` — 获取估值指标
- `compare_peers` — 同行业对比

**输出格式**：
- 营收、利润、现金流及增速
- PE/PB/PS/PEG 等估值指标
- 行业排名和对比数据
- 近期重要财务事件

### 3.3 舆情 Subagent

**输入**：股票代码/公司名称

**可用 Tools**：
- `search_news` — 搜索财经新闻和公告
- `analyze_sentiment` — 情绪分析
- `get_insider_trades` — 大股东/高管增减持

**输出格式**：
- 近期重要新闻和公告列表
- 整体情绪倾向（正面/负面/中性新闻比例）
- 大股东和高管增减持记录
- 是否有重大事件即将发生

## 4. Decision Agent 四步管线

Decision Agent 是系统的核心。它不自己收集数据，只在 Orchestrator 汇总好信息后做决策。决策过程是固定的四步，顺序不可跳过。

### 4.1 第一步：准入检查（硬规则）

通过 `pre_trade_check` Tool 执行，Decision Agent 无法绕过。

检查项：
- 日内交易次数是否达到上限
- 今日亏损是否触发熔断线
- 最大回撤是否进入防守模式（防守模式下禁止新开仓）
- 总仓位是否已达上限
- 标的流动性是否足够

任一项不通过，直接返回拒绝，不进入后续步骤。

### 4.2 第二步：信号生成（Prompt 引导）

Decision Agent 拿到三方信息 + 当前匹配的策略定义，通过推理判断方向。

这一步不是硬编码规则，而是靠 Prompt 引导 Agent 的推理框架：
- 先看策略定义的入场条件，逐条对照当前数据
- 技术面数据是否满足策略要求的形态/指标条件
- 基本面数据是否存在明显矛盾（比如策略说买，但基本面有重大利空）
- 舆情是否有可能改变短期走势的重大事件

输出：买入 / 卖出 / 观望，附带推理过程。

### 4.3 第三步：仓位计算（硬规则）

通过 `calculate_position` Tool 执行。

核心逻辑：**每笔交易最多亏多少 → 反推仓位大小**
- 单笔最大亏损 = 总资金 x 固定比例（如 1%）
- 单股风险 = 入场价 - 止损价
- 理论股数 = 单笔最大亏损 / 单股风险
- 再取以下约束的最小值：
  - 策略定义的单票仓位上限
  - 全局单票仓位上限
  - 剩余可用仓位

止损位越远 → 仓位越小；止损位越近 → 仓位可以更大。风险恒定。

### 4.4 第四步：止盈止损设定（策略定义）

跟着当前匹配的策略走，在进场前就定好：
- 止损位：由策略定义（如 N 倍 ATR、跌破关键支撑位等）
- 止盈位：由策略定义（如目标价、分批止盈规则等）
- 最大持有时间：部分策略有时间止损

所有参数在交易计划中明确，不留"到时候再看"。

## 5. 策略扩展机制

### 5.1 策略定义规范

每个策略是一个 Deep Agents Skill 文件，定义三件事：

**适用行情**：
- 市场状态（趋势上涨/趋势下跌/震荡/高波动/低波动）
- 波动率范围
- 适用行业（可选）

**入场出场条件**：
- 需要满足的技术指标条件
- 需要关注的基本面要素
- 出场触发条件

**风控参数**：
- 单票最大仓位比例
- 止损倍数（如 ATR 的几倍）
- 止盈目标
- 最大持有天数

### 5.2 策略匹配流程

1. Orchestrator 先让技术面 Subagent 判断当前市场状态（趋势/震荡/波动率等）
2. 用 `select_strategy` Tool 根据市场状态匹配适用策略
3. 将匹配到的策略定义注入 Decision Agent 的上下文
4. Decision Agent 在策略框架下做具体决策

### 5.3 新增策略

新增策略 = 在 `skills/` 目录下添加一个 Skill 文件，遵循上述规范。无需修改任何 Agent 代码或 Prompt。

目录结构：
```
skills/
  strategies/
    momentum-breakout/
      SKILL.md        # 动量突破策略定义
    mean-reversion/
      SKILL.md        # 均值回归策略定义
    grid-trading/
      SKILL.md        # 网格交易策略定义
    event-driven/
      SKILL.md        # 事件驱动策略定义
```

## 6. 交易审批与信号推送

### 6.1 默认模式：信号推送

Decision Agent 完成决策后，调用 `send_trade_signal` Tool 推送交易计划。

推送渠道：企业微信 / Telegram / 钉钉（可配置）。

推送内容：
- 交易方向和信号强度
- 建议入场价格
- 建议仓位大小
- 止损位和止盈位
- 关键风险点
- 失效条件（什么情况下这个计划作废）
- 分析摘要

推送不阻塞流程，用户收到后自行决定是否执行。

### 6.2 自动模式：HITL 审批

如果启用自动下单，`place_order` Tool 配置 `interrupt_on`，执行前阻塞等待用户确认。这是唯一需要 HITL 的环节。

可选扩展：高确信 + 无分歧的信号自动执行，低确信的降级为纯信号推送。

## 7. 持久化（Store）

跨会话需要记住的状态：
- **用户风险偏好**：激进/稳健/保守，影响仓位计算和策略匹配
- **当前持仓**：已持有的股票、成本、仓位占比
- **交易记录**：历史交易，用于日内交易计数和亏损计算
- **风控状态**：是否处于防守模式、今日已用额度

通过 Deep Agents 的 `Store` (InMemoryStore 或持久化 Store) 实现。

## 8. Prompt 体系

### 8.1 Orchestrator Prompt

核心要点：
- 你是调度员，收到用户请求后判断需要收集哪些信息
- 用 `write_todos` 拆解步骤，派给对应 Subagent
- 收集完毕后将所有信息交给 Decision Agent
- 你不做分析、不做决策、不给交易建议

### 8.2 信息收集 Subagent Prompt

核心要点（三个 Subagent 共性）：
- 你的职责是收集和整理信息，**只报事实，不给结论**
- 按指定的结构化格式返回数据
- 不要说"建议买入"或"看好"之类的判断性语言
- 如果数据不完整或异常，如实报告，不要补脑

各 Subagent 的差异在于可用 Tools 和输出格式不同。

策略注入：当 Orchestrator 已匹配到策略时，可在派活指令中附加策略专属的数据采集侧重点（如"重点关注成交量配合"）。

### 8.3 Decision Agent Prompt

核心要点：
- 你是决策者，基于收集到的信息做出交易决策
- 严格按四步顺序执行：准入检查 → 信号生成 → 仓位计算 → 止盈止损
- 准入检查和仓位计算必须调用对应 Tool，不能自己算
- 信号生成时，对照当前策略定义的入场条件逐条检查
- 如果信息不足以做出判断，结论应该是"观望"而不是猜测
- 输出完整的交易计划，包含所有必要参数

动态注入：
- 当前匹配的策略定义（从 Skill 加载）
- 用户风险偏好（从 Store 读取）
- 当前持仓状态（从 Store 读取）

### 8.4 策略 Skill Prompt

写给 Decision Agent 看的结构化指令，不是人类文档。格式固定：
- 适用条件（行情状态 + 波动率范围）
- 入场条件（逐条列出，Agent 对照检查）
- 出场条件（逐条列出）
- 风控参数（具体数值）
- 分析侧重点（注入给信息收集 Subagent 的指引）

## 9. 完整交易流程示例

用户："分析一下贵州茅台，最近能不能买？"

1. **Orchestrator** 收到请求，用 `write_todos` 拆解：
   - 收集技术面数据
   - 收集基本面数据
   - 扫描舆情
   - 综合决策

2. **三个 Subagent 并行收集**（Subagent 无状态，一次派活包含完整指令）：
   - 技术面返回：K线数据、指标数值、形态识别、关键价位
   - 基本面返回：财报数据、估值指标、行业排名
   - 舆情返回：新闻列表、情绪倾向、增减持记录

3. **策略匹配**：根据技术面返回的市场状态，匹配到"动量突破"策略

4. **Decision Agent 四步决策**：
   - 准入检查 → 通过
   - 信号生成 → 对照动量策略入场条件逐条检查 → 结论：弱买入
   - 仓位计算 → 基于止损距离反推 → 建议 5% 仓位
   - 止盈止损 → 止损 ATR x 2，分批止盈

5. **推送信号** 到用户配置的渠道

## 10. 技术选型

| 组件 | 选型 | 说明 |
|------|------|------|
| Agent 框架 | Deep Agents (LangChain/LangGraph) | 内置 Subagent + TodoList + HITL + Skills |
| 语言 | Python | Deep Agents 主要支持 Python |
| LLM | Claude Sonnet（Subagent）/ Claude Opus（Decision Agent） | 决策用更强的模型 |
| 行情数据 | AKShare / Tushare | A股数据源 |
| 消息推送 | 企业微信 Webhook / Telegram Bot API | 交易信号推送 |
| 持久化 | Deep Agents Store | 持仓、偏好、交易记录 |
| 策略管理 | Deep Agents Skills | 按需加载，可插拔 |
